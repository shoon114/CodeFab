# 우리가 특히 신경 쓴 예외 케이스들

> 발표 원본 자료. 데모(`docs/presentation/demo/`)에서 직접 재현해서 보여준다.

이 프로젝트는 콘솔에 실제로 입력했을 때의 동작(system test)을 기준으로
검증하는 단계로 넘어오면서, "문법적으로는 맞지만 여러 줄에 걸쳐 입력하면
깨지는" 부류의 버그를 여러 건 찾아서 고쳤다. 아래는 그중 데모로 보여주기
좋은 것들이다.

## 1. 여러 줄에 걸친 제어문 선언 (Allman 스타일)

`if (...)`, `for (...)`, `func name(...)`, `Class Name`처럼 조건/선언부와
`{` 시작 지점이 다른 줄에 오면, REPL이 한 줄씩 입력을 받아 "완결된
문장인지" 판단하는 로직이 이를 완성된 문장으로 착각해서 곧바로 실행을
시도하는 문제가 있었다.

```javascript
func add(a, b)
{
  return a + b;
}
print add(2, 3);   // 5
```

이 코드를 한 줄씩 입력하면, `func add(a, b)`만 보고 "본문이 없다"는
에러를 내며 버퍼를 비워버리고, 이어서 입력되는 `{ ... }`는 함수 선언과
분리된 채 독립된 블록으로 실행되어(내부의 `return`이 함수 밖에서
실행되는 것으로 취급되어 별도 에러까지 발생) 함수가 아예 정의되지 않는
결과로 이어졌다. `if`/`for`/`func`/`Class` 네 가지 모두에서 같은 패턴의
버그를 찾아 동일한 방식으로 고쳤다.

## 2. `else` vs `else if`를 구분해야 하는 대기 로직

```javascript
if (a > 3) print "x";
else if (b > 1) print "y";
else print "z";
```

"버퍼에 `else`가 있는지"만으로는 부족하다 — `else if`도 `else`로 카운트
되기 때문에, 3단 체인의 마지막 `else`를 기다리는 도중에 앞의 `else if`
때문에 "이미 결론이 났다"고 오판해서 뒤이은 순수 `else`가 짝을 잃고
구문 오류가 나는 문제가 있었다. "체인을 끝내는 순수 `else`(뒤에 `if`가
없는 else)가 이미 나왔는지"로 판단 기준을 바꿔서 해결했다.

여기서 한 걸음 더 들어가면: 조건과 body가 한 줄에 합쳐진 스타일
(`if (...) 문장; else if (...) 문장; else 문장;`)에서는 "이미 한 번
기다렸는지"를 단순 bool 하나로만 추적하면 또 다른 함정에 빠진다.
`else if`의 조건과 body가 한 번에 통째로 들어오면서 "새로 대기해야 할
지점"인데도 "예전에 이미 기다렸던 상태"로 착각해 성급하게 실행해버리는
것이다. "마지막으로 대기를 시작했을 때 버퍼에 있던 else 개수"를
정수로 추적해, else 개수가 실제로 늘어났을 때만(=체인이 진짜로 한 칸
더 연장됐을 때만) 새로 대기하도록 바꿔서 해결했다. 겉보기엔 사소한
입력 스타일 차이가 완전히 다른 버그로 이어질 수 있다는 걸 보여주는
사례다.

## 3. `for` 문의 스코프 누수

```javascript
for (var i = 0; i < 3; i = i + 1) { print i; }
print i;   // 에러: i는 for문이 끝나면 사라져야 한다
```

`{}` 블록이 끝나면 그 안의 지역 변수가 사라지는 건 잘 지켜지고 있었지만,
`for`의 **init에서 선언된 변수**(`var i`)는 별도 스코프 없이 바깥
스코프에서 바로 실행되고 있어서 반복문이 끝난 뒤에도 계속 남아있었다.
`{}` 없는 단일 문장 body 안에서 선언된 변수도 마찬가지 문제가 있었다.
둘 다 "for 문 전체를 하나의 스코프로 감싼다"는 원칙으로 통일해서
고쳤다 — 이때 `ExecutorUnit`(런타임)과 `CheckerUnit`(정적 스코프 거리
계산)의 스코프 프레임 수를 정확히 같이 맞춰야 했다는 점이 까다로웠다.

## 4. 함수는 호출자의 지역 변수를 볼 수 없어야 한다

```javascript
func tryAccess() { return localOnly; }
{ var localOnly = 5; print tryAccess(); }   // 에러: 함수는 호출자 스코프를 못 봄
```

함수는 전역 변수는 참조할 수 있어야 하지만, 호출자가 우연히 같은 이름의
지역 변수를 갖고 있다고 해서 그걸 몰래 참조할 수 있으면 안 된다. 이
경계를 명확히 테스트로 고정해뒀다.

## 5. 정적 오류가 있으면 실행을 멈춰야 한다

```javascript
func add(a, b) { return a + b; }
print add(1);   // 인자 개수 불일치
```

`CheckerUnit`이 정적 오류(인자 개수 불일치 등)를 이미 보고했는데도
`ExecutorUnit`이 계속 실행을 이어가서, 같은 문제에 대해 "정적 오류
메시지"와 "런타임 오류 메시지"가 중복으로 출력되는 문제가 있었다.
정적 오류가 있으면 실행 자체를 하지 않도록 고쳐서, 사용자에게 혼란을
주는 이중 에러 메시지를 없앴다.

## 6. 테스트 하네스 자체의 신뢰성

콘솔에 대화형 프롬프트(`>>> `)를 추가하는 기능이 들어오면서, 그 프롬프트가
stderr로 출력된다는 이유만으로 회귀 테스트의 "에러 발생 여부" 판정이
실제 에러와 무관하게 항상 통과로 나오게 되는 문제가 있었다. 기능
코드뿐 아니라 **그 기능을 검증하는 테스트 하네스 자체도 새 기능 때문에
조용히 무력화될 수 있다**는 걸 발견하고 즉시 고친 사례다.
