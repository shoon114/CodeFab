# 개발 프로세스: TDD, System Test, 협업

> 발표 원본 자료.

## 1. 1단계 — TDD로 각 모듈 API 검증

프로젝트 초반부터 각 파서/`CheckerUnit`/`ExecutorUnit` 모듈은 유닛 테스트
(gmock/gtest) 기준으로 개발됐다.

- 사이클: **설계 제안 → 사용자/팀 확인 → TC 작성(RED) → 구현(GREEN) →
  빌드/테스트 확인 → 커밋**.
- 다른 파서에 대한 의존성은 항상 `MockStatementParser`로 stub해서
  격리 검증한다(실제 구현체에 의존하지 않음).
- 전역 싱글턴 레지스트리(`StatementParserRegistry`)를 다루는 TC는
  fixture `SetUp()`/`TearDown()`에서 반드시 값 캡처 + 원래 상태 복원을
  지켜야 한다 — 이 패턴을 빠뜨려 발생한 실제 크래시 사례가 있었고, 이후
  전수 점검으로 잡아냈다.

## 2. 전환점 — System Test 단계

유닛 테스트로 각 모듈의 API 수준 정확성이 어느 정도 확보된 뒤부터는
검증 방식을 바꿨다:

> 새 기능/버그 수정은 유닛 테스트를 새로 작성하는 대신, **실제 사용자가
> 콘솔에 입력하는 코드 -> 최종 실행 결과**를 검증하는 system test로
> 진행한다.

- `.claude/skills/system-test/run.ps1`에 지금까지 검증한 모든 콘솔
  입력/기대 출력을 케이스로 계속 누적한다.
- **Release 빌드 1회 + 실행 N회** 구조라, 케이스가 수십 개로 늘어나도
  다시 컴파일하지 않고 몇 초 안에 전체 회귀를 확인할 수 있다.
- 기존 유닛 테스트(TC)는 폐기되지 않는다 — 새 기능을 system test로
  검증하더라도, 커밋 전에는 항상 기존 유닛 테스트 스위트 전체가 PASS
  하는지 함께 확인한다. system test는 유닛 테스트를 "대체"하는 게 아니라
  "새 개발의 검증 방식을 바꾼 것"이다.
- 에러 케이스는 정확한 메시지 문자열을 요구하지 않고 "에러가 발생했는가"
  만 확인하는 `ExpectError` 타입으로 별도 관리해, 메시지 문구가 나중에
  바뀌어도 테스트가 깨지지 않게 했다.

## 3. Git 워크플로우

- 기능/버그 수정 단위로 브랜치를 나누고, PR 단위로 squash-merge한다.
- 브랜치를 rebase할 때, 이미 squash-merge된 브랜치 위에서 다시
  작업하려면 일반 `git rebase master`(patch-id 기반 자동 스킵)가 아니라
  `git rebase --onto master <old-base-commit> <branch>`로 진짜 새
  커밋만 replay한다 — 그렇지 않으면 이미 merge된 커밋들이 통째로 다시
  충돌을 일으킨다.
- 논리적으로 구분되는 케이스/수정마다 커밋을 분리해서, 리뷰어가 diff
  단위로 "이 변경이 왜 필요한지"를 바로 알 수 있게 한다.

## 4. 협업 — 코드 리뷰로 잡아낸 문제들

여러 명이 같은 코드베이스의 서로 다른 모듈을 동시에 발전시키면서,
PR 리뷰/버그 리포트를 통해 아래와 같은 문제들이 잡혔다(팀 전체 기준,
아래는 커밋 로그에 남아있는 대표 사례):

- `CheckerUnit`이 함수 중복 선언을 검사하지 않던 문제
- `CheckerUnit`이 메서드/클래스 생성자 호출을 미정의 함수로 오판하던 문제
- RAII 기반으로 스코프 생성/소멸 시점을 통일한 리팩토링
  (예외가 중간에 던져져도 스코프가 항상 정리되도록)
- `if`문의 스코프 처리 방식을 팀 논의를 거쳐 재조정

이런 이슈들은 한 사람이 전체 코드를 다 알지 못해도, "이 모듈은 이런
계약을 지킨다"는 걸 유닛 테스트/system test로 명시해뒀기 때문에 리뷰
과정에서 비교적 빠르게 드러나고 고쳐질 수 있었다.
