Architecture

파서: Strategy + Registry

AssemblerUnit::Parse()는 statement 타입을 switch/if-else로 분기하지 않는다. 모든 파서는 IStatementParser 하나로 통일되고, 전역 레지스트리가 실행 시점에 알맞은 파서를 찾아 위임한다.

자기 등록새 파서 추가 시 AssemblerUnit은 수정 불필요 — .cpp에 registrar 한 줄만 추가.

생성자 주입 금지하위 파서는 항상 Resolve()로 실행 시점에 얻어온다.

tokenList[pos].type
  -> Registry.Resolve(type)
  -> IStatementParser::Parse(...)
  -> SyntaxNode

// 파서 자기 등록 (VarDeclareParser.cpp)
namespace {
  StatementParserRegistrar<VarDeclareParser>
    registrar(TokenType::KwVar);
}
Architecture

AST 순회: GoF Visitor 패턴

CheckerUnit/ExecutorUnit이 NodeType을 switch/if-chain으로 직접 분기하던 방식을 더블 디스패치로 교체했다.

  • 각 구체 노드(IfStmtNode, ClassDeclStmtNode …)가 Accept()에서 자신의 정확한 타입으로 Visit()을 호출
  • NodeVisitor의 기본 구현은 자식을 그대로 순회(Traverse) — 관심 있는 노드만 override하면 됨
  • 새 노드 타입이 추가돼도 관심 없는 기존 Visitor는 코드 수정 불필요
Architecture

스코프 / 클래스 모델

스코프

ScopeGuard(RAII)가 EnterScope/ExitScope를 보장 — 예외(return)가 던져져도 스코프가 항상 정리된다.

CheckerUnit의 정적 스코프 스택과 ExecutorUnit의 런타임 스코프 스택은 프레임 수가 정확히 일치해야 한다.

클래스/상속

필드는 unordered_map에 동적으로 보관, 메서드는 부모 체인을 타고 탐색.

Super.method()는 인스턴스의 실제 클래스가 아니라 정의된 클래스의 부모부터 탐색해야 무한 재귀를 피한다.

Process

TDD → System Test

1단계 — 각 파서/Checker/Executor를 gmock/gtest 유닛 테스트로 검증 (설계 제안 → 확인 → RED → GREEN → 커밋).

전환점 — API 정확성이 확보된 뒤로는, 새 기능/버그 수정을 콘솔 입력 → 실행 결과 기준의 system test로 검증.

  • .claude/skills/system-test/run.ps1에 케이스 누적, Release 빌드 1회 + 실행 N회로 수십 개 케이스도 몇 초 안에 회귀 확인
  • 기존 유닛 테스트는 폐기 X — 커밋 전 항상 전체 스위트 PASS 확인
  • ExpectError 타입: 메시지 문구가 아니라 "에러 발생 여부"만 확인

  • 1여러 줄에 걸친 제어문 선언 (Allman 스타일)

전부 "문법적으로는 맞지만 여러 줄/입력 스타일에 따라 깨지는" 부류 — system test 전환 이후 발견/수정.

Edge Cases · Deep Dive

같은 문법, 다른 입력 스타일 → 다른 버그

// 스타일 A: 조건과 body가 다른 줄 — 정상
if (a > 3)
{ print "x"; }
else if (b > 1)
{ print "y"; }
else
{ print "z"; }

// 스타일 B: 조건+body가 한 줄에 합쳐짐 — 한때 "Unexpected token" 에러
if (a > 3) print "x";
else if (b > 1) print "y";
else print "z";

"이미 한 번 기다렸는지"를 bool 하나로 추적하면, 스타일 B에서 새로 대기해야 할 지점을 "예전에 기다렸던 상태"로 착각한다. "대기 시작 시점의 else 개수"를 정수로 추적해, 개수가 실제로 늘어났을 때만 다시 기다리도록 바꿔서 해결했다.

기능 데모 (약 5분)

docs/presentation/demo/의 파일 순서대로 Release 콘솔에 직접 입력하며 진행한다.

  1. 01 — 인터랙티브 디버그 모드 (step/next/watch/inspect)

  3. 03 — 클래스 (상속, 오버라이딩, Super 호출, 다형성)

  5. 05 — 함수 스코프 격리 (호출자 로컬 변수 은닉, 정적 오류 시 실행 중단)
